  1. RESUMEN DEL SISTEMA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  CHASKI AI es una plataforma web para digitalizar y optimizar la gestión
  operativa de la Asociación de Transportistas Virgen de Fátima, que opera la
  ruta Juli–Puno (98.5 km) con más de 60 vehículos socios.

  PROBLEMA QUE RESUELVE:
    · Control de cola de salida hecho a mano (conflictos entre conductores)
    · Manifiestos de viaje en papel (se pierden, ilegibles, lentos)
    · Sin visibilidad GPS de la flota en tiempo real
    · Sin reportes de ingresos ni desempeño de conductores

  TECNOLOGÍAS USADAS:
    · Frontend  : HTML5 + CSS3 + JavaScript Vanilla
    · Backend   : Node.js + Express.js
    · Base datos: PostgreSQL
    · Tiempo real: Socket.io (WebSockets)
    · Mapas     : Leaflet.js
    · Animación : Three.js (landing page), GSAP
    · Auth      : JWT + bcryptjs
    · IoT       : Teltonika FMC130 (dispositivo GPS, protocolo Codec 8 TCP)
    · IA        : Claude API de Anthropic (Opción C, por implementar)

  RUTA DE LA ASOCIACIÓN: Juli (Chucuito, Puno) → Puno · 98.5 km · S/ 7.00

  TABLAS Y SU PROPÓSITO:

  associations        → Agrupa asociaciones (Virgen de Fátima, TipCar, etc.)
  companies           → Empresas de transporte dentro de cada asociación
  vehicle_types       → Tipos de vehículo (HiAce 12p, Sprinter 19p, Master 15p)
  users               → Usuarios del sistema (admin, driver, supervisor) + JWT auth
  vehicles            → Flota de vehículos con campo gps_device_id (FMC130)
  drivers             → Conductores socios vinculados a usuario y vehículo
  routes              → Rutas definidas (Juli→Puno, Puno→Juli · 98.5 km · S/7.00)
  exit_queue          → Cola de salida diaria FIFO con estados
  manifests           → Manifiestos de viaje (cabecera: vehículo, conductor, ruta)
  manifest_passengers → Pasajeros de cada manifiesto (DNI, asiento, pago)
  trips               → Viajes realizados (velocidad media, máxima, duración)
  gps_positions       → Telemetría GPS del FMC130 (lat, lon, speed, altitud, sat)
  speed_alerts        → Alertas de velocidad (>90 km/h) con severidad y ACK
  route_alerts        → Alertas de desvío de ruta, paradas no programadas
  system_alerts       → Notificaciones generales del sistema para el panel admin

  JERARQUÍA DE DATOS:
    Asociación → Empresa → Vehículo → Driver → Trip → Manifest → Passengers
                                    ↑
                            gps_positions (FMC130)
                            speed_alerts
                            route_alerts

  ÍNDICES CREADOS (rendimiento):
    · gps_positions  → idx_gps_vehicle_time (vehicle_id + recorded_at DESC)
    · gps_positions  → idx_gps_trip
    · vehicles       → idx_vehicles_company, idx_vehicles_code
    · manifests      → idx_manifests_vehicle, idx_manifests_date
    · trips          → idx_trips_vehicle, idx_trips_date
    · exit_queue     → idx_exit_queue_date
    · speed_alerts   → idx_speed_alerts_vehicle

  6. IOT — INTEGRACIÓN TELTONIKA FMC130
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ESTADO: Base de datos lista. Servidor TCP PENDIENTE (Hito 4, 16/05–05/06/2026)

  DISPOSITIVO: Teltonika FMC130
    · Conectividad: 4G LTE Cat 1 / 3G / 2G (fallback automático)
    · GNSS: GPS + GLONASS + BeiDou + Galileo (precisión 2–5 metros)
    · Protocolo: Codec 8 sobre TCP (binario comprimido)
    · Alimentación: 10–30V DC (batería del vehículo)
    · Memoria: 128 MB store & forward (guarda datos sin cobertura)
    · Acelerómetro: triaxial (detecta frenazos, aceleración agresiva, impactos)

  FLUJO DE DATOS:
    [FMC130 en vehículo]
         │ TCP puerto 5027
         ↓
    [backend/tcp-server.js]   ← decodifica Codec 8
         │
         ├─ INSERT en gps_positions (lat, lon, speed, heading, accel xyz)
         ├─ Si speed > 90 → INSERT en speed_alerts
         ├─ Si accel anómalo → llama a anomalyService.js (Opción A)
         └─ Socket.io emit('gps:update') → mapa Leaflet admin en tiempo real

  CONFIGURACIÓN DEL DISPOSITIVO (con Teltonika Configurator — Windows):
    · Server IP  : IP del VPS DigitalOcean (ej. 142.93.12.34)
    · Server Port: 5027
    · Protocol   : Codec 8
    · Min Period : 10 segundos (en movimiento) / 60 segundos (estático)
    · Min Distance: 50 metros
    · Acelerometer: habilitado, sensibilidad media

  ESTRUCTURA DEL PAQUETE CODEC 8 (simplificado):
    [2 bytes: zeros] [4 bytes: data length] [1 byte: Codec ID=0x08]
    [1 byte: AVL count] [ AVL records... ] [1 byte: AVL count]
    [4 bytes: CRC-16]

    Cada AVL record contiene:
      · Timestamp (8 bytes, Unix ms)
      · Lat, Lon (4 bytes cada uno, factor 10^7)
      · Altitude, Speed, Heading
      · IO Elements: incluye acelerómetro (ID 17: X, 18: Y, 19: Z)

  PILOTO: Se inicia con 1 dispositivo FMC130 (S/300). Luego se escala a 60.

  7. OPCIÓN A — DETECCIÓN DE ANOMALÍAS EN CONDUCCIÓN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ESTADO: POR IMPLEMENTAR (después de que el servidor TCP esté funcionando)
  ARCHIVO: backend/services/anomalyService.js

  QUÉ DETECTA:
    · Frenazo brusco      → acelerómetro eje Z < -0.4g
    · Aceleración agresiva → acelerómetro eje Z > 0.4g
    · Curva agresiva       → acelerómetro eje X o Y > 0.3g
    · Velocidad excesiva   → speed > 90 km/h (ya en BD como speed_alerts)
    · Patrón anómalo       → Z-score > 2 desviaciones estándar del historial

  ALGORITMO Z-SCORE (sin librerías externas):
    1. Obtener historial de los últimos 30 días del conductor
    2. Calcular media (μ) y desviación estándar (σ) por métrica
    3. Para cada nueva lectura: Z = (valor - μ) / σ
    4. Si |Z| > 2.0 → evento anómalo → registrar en speed_alerts con
       severity = 'warning' (Z > 2) o 'critical' (Z > 3)

  PUNTAJE DE CONDUCCIÓN (0-10 por conductor):
    · Base 10 puntos
    · -0.5 por cada alerta 'warning'
    · -1.0 por cada alerta 'critical'
    · Mostrado en admin/drivers.html y en admin/reports.html

  OUTPUT EN EL SISTEMA:
    · Panel admin → sección "Alertas" con conductor, tipo, severidad, ubicación
    · Panel admin → tabla conductores con columna "Puntaje de conducción"
    · [Opción C] → Claude AI puede generar un resumen narrativo de las anomalías

  VALOR PARA LA TESIS:
    Usa el acelerómetro triaxial del FMC130 (dato técnico real del charter).
    Es el componente de "Inteligencia" que justifica el nombre CHASKI AI.
    Defensa ante jurado: "Sistema de scoring de conducción basado en análisis
    estadístico de telemetría IoT con detección de anomalías por Z-score"
